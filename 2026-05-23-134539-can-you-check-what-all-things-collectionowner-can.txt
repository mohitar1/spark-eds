⏺ Great question — it's actually a hybrid model, not purely RBAC. Here's the
  breakdown:

  ---
  Access Control Model: RBAC + ABAC Hybrid

  RBAC Layer — Global Access

  Global roles assigned at login from a permissions config sheet:

  admin | employee | contingent-worker | agency | bottler

  These control: who can log in, which pages they see, and whether they have
  base access at all.

  Collection-level roles (owner/editor/viewer) are also RBAC-style flat roles.

  ---
  ABAC Layer — Asset & Collection Access

  This is where it gets attribute-based. Access decisions are computed at 
  request time by comparing user attributes against resource attributes:

  ┌─────────────┬─────┬───────────────────────────────────┬─────────────────┐
  │    User     │ vs  │        Resource Attribute         │    Decision     │
  │  Attribute  │     │                                   │                 │
  ├─────────────┼─────┼───────────────────────────────────┼─────────────────┤
  │             │     │                                   │ Deny if brand   │
  │ user.brands │ vs  │ assetMetadata.tccc:brand          │ not in user's   │
  │             │     │                                   │ allowed brands  │
  ├─────────────┼─────┼───────────────────────────────────┼─────────────────┤
  │ user.countr │ vs  │ assetMetadata.tccc:intendedBottle │ Deny if country │
  │ ies         │     │ rCountry                          │  doesn't match  │
  ├─────────────┼─────┼───────────────────────────────────┼─────────────────┤
  │ user.custom │     │ assetMetadata.tccc:intendedCustom │ Deny if         │
  │ ers         │ vs  │ ers                               │ customer not in │
  │             │     │                                   │  user's list    │
  ├─────────────┼─────┼───────────────────────────────────┼─────────────────┤
  │             │     │ tccc:acl (per-collection          │ Allow only if   │
  │ user.email  │ vs  │ metadata)                         │ in owner/editor │
  │             │     │                                   │ /viewer list    │
  └─────────────┴─────┴───────────────────────────────────┴─────────────────┘

  The key function is buildAssetAuthClauses() in dm.js — it dynamically
  constructs search filters on every request based on the calling user's
  attributes. No hardcoded permission matrix.

  ---
  Why Collection ACLs are ABAC-like (not purely RBAC)

  In pure RBAC, roles are global ("user X is an Editor"). Here, each collection 
  instance has its own ACL stored in its metadata:

  tccc:acl → {
    tccc:assetCollectionOwner: "alice@co.com",
    tccc:assetCollectionEditor: ["bob@co.com"],
    tccc:assetCollectionViewer: ["carol@co.com"]
  }

  Alice might be an Owner of Collection A but a Viewer of Collection B. The role
   is a property of the resource instance, evaluated per-request — that's the
  ABAC characteristic.

  ---
  Summary

  ┌─────────────────┬───────────┬───────────────────────────────────────────┐
  │      Layer      │   Model   │                   Where                   │
  ├─────────────────┼───────────┼───────────────────────────────────────────┤
  │ Login / page    │ RBAC      │ Global roles from config sheet            │
  │ visibility      │           │                                           │
  ├─────────────────┼───────────┼───────────────────────────────────────────┤
  │ Collection      │ ABAC-like │ Per-collection ACL in metadata            │
  │ access          │           │                                           │
  ├─────────────────┼───────────┼───────────────────────────────────────────┤
  │ Asset search    │ ABAC      │ buildAssetAuthClauses() — user attrs vs   │
  │ filtering       │           │ asset attrs                               │
  ├─────────────────┼───────────┼───────────────────────────────────────────┤
  │ Asset metadata  │ ABAC      │ enforceAssetMetadataAuthorization()       │
  │ fetch           │           │                                           │
  └─────────────────┴───────────┴───────────────────────────────────────────┘

  So: RBAC for "who gets in the door", ABAC for "which specific
  assets/collections they can see and touch." The ABAC layer is quite
  sophisticated — driven by brands, countries, and customer attributes resolved
  dynamically at query time, not at role-assignment time.
