  ---
  The shift: from "feature modules" to "feature generation"

  Two ways to deliver advanced features:

  (A) Pre-build everything, toggle per tenant. Boilerplate ships
  rights/reports/notifications as code, customer flips flags. → Huge
  boilerplate, most of it dead for any given tenant, and "add a new feature
  nobody pre-built" still needs a human. This is what I was drifting toward.
  It's wrong for your goal.

  (B) Ship a minimal capable core + agents that generate features on request.
  Boilerplate ships only the irreducible portal (search, asset detail, download,
  collections, share). Everything else — including things no one anticipated —
  is built by skills when a customer asks. → Small boilerplate, infinite feature
  surface, genuinely agentic.

  You want (B). That makes the boilerplate's job not "contain every feature" but
  "be a substrate that agents can reliably extend." The design effort moves
  into two things: what the substrate must expose so generation is reliable, and
  what the agent must know to generate correctly.

  ---
  What makes feature-generation reliable (the substrate)

  An agent can only reliably build a feature if the boilerplate is patterned and
  self-describing. Concretely, the boilerplate must ship:

  1. A "golden path" feature — one fully-built reference feature (say,
  collections) implemented exactly the way every future feature should be: block
  + worker route + DM call + tests + docs. Agents build new features by analogy
  to it. This is the single most important thing — agents extend well when
  there's a canonical example to mirror.
  2. Stable extension points the agent writes into, not around:
    - Worker: a route-registration convention (drop a handler in
  api/<feature>.js, register in one place) so adding an endpoint is mechanical,
  not surgery on index.js.
    - Blocks: the EDS block contract already is this (decorate(block) + folder)
  — agents already build blocks via your existing building-blocks skill.
    - Auth/roles/config: a feature declares its required role + config keys in
  one manifest, rather than hand-wiring access checks.
  3. A capability map of the DM OpenAPI (see next section) — so the agent knows
  which API operation backs the feature it's asked to build.
  4. A feature contract / template — a checklist every generated feature
  satisfies (block, route, DM binding, role, i18n, test, doc entry). The skill
  enforces it so generated features look hand-written and don't rot.

  The design insight: the boilerplate's value isn't its feature count — it's how
  cheaply an agent can add the (N+1)th feature. That's what you optimize.

  ---
  How the DM OpenAPI lives inside the agents/skills

  This is the part you're specifically pushing on, and it's the crux. The DM
  OpenAPI is the feature substrate — almost every distribution-portal feature is
  "call this DM operation + render it." So the agent must treat the spec as a
  first-class, queryable asset, not lore in its head.

  Concretely, three layers:

  1. The spec ships in the repo, pinned. Vendor the OpenAPI YAML (the redocly
  bundle) into the boilerplate at a known path, version-pinned. Now it's ground
  truth the agent reads — not a URL it might hallucinate.

  2. A dm-api skill that is the only sanctioned way to touch the API. When a
  customer asks for a feature, the building skill consults dm-api to answer:
  which operation backs this? what are its params, auth, response shape? does it
  already have a typed client method in the worker's DM client? The skill maps
  intent → operation:

  ┌────────────────────────┬────────────────────────────────────────────────┐
  │   Customer asks for…   │        DM operation the skill binds to         │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ "bulk download /       │ POST /archives → poll /archives/{id}/status →  │
  │ download all"          │ GET /archives/{id}/files/{name}                │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ "collections"          │ /collections, /collections/{id}/items,         │
  │                        │ jobs/{id}/status                               │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ "download in a         │ GET /{id}/as/{seo}.{fmt}, /imagePresets,       │
  │ specific format/size"  │ /renditions                                    │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ "video preview"        │ /{id}/play, /{id}/manifest.{fmt}               │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ "smart crops"          │ /{id}/smartCrops                               │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ "usage analytics"      │ asset_downloaded / asset_search events         │
  └────────────────────────┴────────────────────────────────────────────────┘

  So "add bulk download" becomes a known recipe: the dm-api skill returns the
  archive flow contract, the build skill generates the worker route + the block
  + the poll-and-download UX by analogy to the golden-path feature. The agent
  isn't inventing — it's binding a request to a documented operation and
  instantiating a pattern.

  3. A generated typed client. Because we have the spec, the DM client methods
  in the worker can be generated/checked from it — so when the agent adds a
  feature it calls a real, spec-backed method, and if Adobe revs the API the
  skill flags drift.

  This is the answer to "how are you adding DM OpenAPI in agents": the spec is 
  vendored + a dm-api skill turns it into intent→operation→contract, and every
  feature-building skill is required to go through it. The DM API stops being
  something the agent vaguely knows and becomes something it looks up and binds 
  to.

  ---
  The customer's view (not rings — a request→change pipeline)

  From the customer's side there are no rings, just kinds of requests, each
  mapped to a skill:

  Customer says…: "Spin up our portal"
  Skill: new-portal
  What it does: Generate tenant config: domains, IDP, CORS, env, branding shell,

    roles
  DM API involved?: sets the delivery/env binding
  ────────────────────────────────────────
  Customer says…: "Make it look like our brand"
  Skill: brand
  What it does: Logo, fonts, color tokens across blocks + email
  DM API involved?: no
  ────────────────────────────────────────
  Customer says…: "Add bulk download / collections / video / formats…"
  Skill: add-feature
  What it does: Bind request → DM operation via dm-api, generate block + route +

    tests by analogy to golden path
  DM API involved?: yes, central
  ────────────────────────────────────────
  Customer says…: "Add a rights/approval/reporting workflow"
  Skill: add-feature (app-layer variant)
  What it does: Generate worker route + KV/D1 store + block; no DM op, uses
    worker extension points
  DM API involved?: no (app-layer)
  ────────────────────────────────────────
  Customer says…: "Restrict who sees what"
  Skill: access
  What it does: Generate roles + metadata-namespace + access rules
  DM API involved?: gates DM proxy/ACL
  ────────────────────────────────────────
  Customer says…: "Author the landing pages"
  Skill: existing EDS skills
  What it does: Your current page-import, building-blocks, etc.
  DM API involved?: no

  So "advanced features built on demand" = the add-feature skill, which has two
  arms: DM-backed (goes through dm-api, e.g. bulk download) and app-layer (uses
  worker storage/extension points, e.g. rights workflow). Both instantiate the
  same golden-path contract so output is consistent.

  ---
  So the boilerplate, restated for your goal

  A small, self-describing portal substrate, shipping:
  - the irreducible working portal (search, asset detail, download, collections,
  share) as the golden-path reference,
  - the worker spine with stable, documented extension points,
  - the vendored DM OpenAPI + a dm-api skill that turns the spec into
  intent→operation→contract,
  - a feature contract every generated feature must satisfy,
  - a skill set keyed to customer request types (new-portal, brand, add-feature,
  access) layered on top of your existing EDS authoring skills.

  The boilerplate is deliberately not feature-complete. Its power is that an
  agent, holding the DM spec and the golden-path pattern, can add the feature a
  customer asks for — anticipated or not — and have it come out looking native.
